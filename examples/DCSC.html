Doors Controlling System Controller (DCSC)

This document specifies the Doors Controlling System Controller (DCSC) device. It covers the device architecture, the components and their state machines, the interfaces between them, the handlers wrapping external resources, and the messages and functions used throughout. Cross-references are clickable; hover the underlined identifiers to navigate to their definitions.

Device specification

The Doors Controlling System Controller (DCSC) is a device that manages the overall operation of the Door Controlling System (DCS) by coordinating the activities of its components and providing secure access control. The Controller's responsibilities fall into six areas.

Door state management

  • Detection. The Controller detects connection and disconnection of doors via the Sensing interface.
  • State query. It queries each door's open/closed and locked/unlocked state through DoorController and updates its internal record accordingly.
  • State monitoring. Door states are monitored continuously so that any change is observed and handled.

Command handling

  • Lock and unlock. Lock and unlock commands arrive from the backend and are executed by forwarding to the relevant DoorController.
  • Tag management. Commands to add or remove access tags are processed by the Controller and forwarded to the appropriate DoorController.
  • Remote operations. The Controller exposes door management to the Administrator via the backend.

Event processing

  • State-change events. Door open/close and lock/unlock events are processed and propagated.
  • Errors and alarms. Error messages and alarm signals from doors are handled and reported.

Interface management

  • Backend. The Controller maintains a connection to the backend, reporting door-state, lock-state, and alarm changes, handling backend error messages, and keeping the backend and the doors in sync.
  • Database. The Controller reads and writes settings and event records via the configuration database, so that system history and configuration are persisted.

System initialisation and configuration

  • Initialisation. At startup the Controller detects all doors, retrieves configuration, and establishes connections with the backend and the other components.
  • Configuration. Database initialisation, backend URL retrieval, and event storage are handled here.

Administrative functions

  • Database management. The Controller supports the Administrator in initialising the database, adjusting system settings, and recording events.
  • Access control. Only authorised tags are accepted, keeping access to the doors restricted.

Together these responsibilities give the DCS Controller end-to-end control over door access, monitoring, and administration.

Administrator

Context

Description

Initializes and exchanges information with the database.

Behaviour

When the Administrator initializes, it needs to also initialize the Configuration Database by calling its ConfigurationDatabase:initDatabase() function. When the database is initialized, the Config:InitializedInd message is received.

When the Configuration Database is initialized, the Administrator can receive the Admin:BackendUrlReq which is a request to retrieve the backend URL. As a result it calls the ConfigurationDatabase:getBackendUrl(). When the database extracted the backend URL, the Administrator received the Config:BackendUrlInd message. This is forwarded by sending the Admin:BackendUrlRsp message.

The Administrator converts the messages on the Admin interface to a proper event description and adds this to the database by calling the ConfigurationDatabase:storeEvent() function. When the database has properly stored the event, the Config:EventStoredInd is received. Only one event can be stored each time, so storing any intermediate events is postponed.

Initialization

As stated, when the Administrator initializes, it needs to also initialize the Configuration Database. Therefore, the default initialize() method must be overridden to call theConfigurationDatabase:initDatabase() function.

Device Controller

Context

Description

Coordinate the overall device, manage door states, and interface with the backend and handle commands from the backend such as locking/unlocking doors, and adding/removing tags.Behavior

Beaviour

At the start, first all doors to be discovered need to be awaited.

The detection of a door is received via the Control:DoorDetectedInd message.

After all doors are detected, the device is ready to connect to the backend via the BackendInterface. However, in order to do that, it requires the URL of the backend. This can be retrieved via the Admin:BackendUrlReq message. When the expected Admin:BackendUrlRsp response is received, the BackendInterface can be instructed to connect by calling its BackendInterface:connect() function.

When the backend is connected, the Commands:ConnectedInd is received. With the connection established, the doors can be released for operation. This is done by sending a Control:ReleaseDoorReq message to each instance of the Control interface.

After this is done, the system is ready to receive commands from the backend via the Commands interface:

When the state of a door changes, the Control:DoorStateChangedInd is received. Likewise, when the door is locked or unlocked, the Control:DoorLockChangedInd is received. These are forwarded to the Backend Interface by calling its BackendInterface:doorStateChanged() and BackendInterface:doorLockChanged() functions respectively.

Any state change needs to be supplied to the database as well to keep the history. This is done by sending the Admin:DoorStateChangedInd and Admin:DoorLockChangedInd respectively.

When an alarm or error occurs with one of the doors, the Control:ErrorInd or Control:AlarmInd messages are received respectively. These are forwarded to the backend by calling the BackendInterface:error() and BackendInterface:alarm() functions.

Door Controller

Context

Description

Monitors and controls the state and lock of the door, and handles sensor data and ensure correct responses to door events (e.g., opening, closing, locking).

Behaviour

When the Door Controller starts, it needs to wait for a door to be detected and assigned to it. This happens when the Sensing:DoorDetectedInd is received. The id of the door contained in the message is then stored in the doorId state variable.

However, although the door is detected, its door-sensor is not enabled yet, i.e. the door is not released for operation. This can be done when the Control:ReleaseDoorReq is revived. Upon receiving this message, the Door Controller can first enable the door sensor by calling the SensorHandler:enableDoorSensor() function.

Consequently, the door sensor will respond with its current status via the Sensing:DoorSensorInd and as a result the Control:ReleaseDoorRsp is sent.

Only when the door is closed will the lock sensor be operational. Thus, in that case, the lock sensor can be triggered to (re)send its state by calling the SensorHandler:enableLockSensor() function.

When the door is opened, a timer needs to be started to detect when the door is opened too long.

In order to lock or unlock the door, the Control:SetLockInd message is used. This command can only be handled when the door is closed. Otherwise, the Control:ErrorInd message is replied.

No explicit reply is required for the Control:SetLockIndmessage because the result of the operation is received via the Sensing:LockSensorInd message.

When a door is opened or closed, the Sensing:DoorSensorInd message is received which can be forwarded via the Control:DoorStateChangedInd message.

When the door is locked, tags can be added or removed from the door’s internal tag administration. Tags are stored at the door itself to allow a door to operate even if the network is down. When it is tried to add or remove tags when the door is not closed, the Control:ErrorInd message is sent.
