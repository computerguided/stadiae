Wireless Interactive Networked Display (WIND)

This document specifies the Wireless Interactive Networked Display (WIND) device. It covers the device architecture, the components and their state machines, the interfaces between them, the handlers wrapping external resources, and the messages and functions used throughout. Cross-references are clickable; hover the underlined identifiers to navigate to their definitions.

Device specification

The device is a Wireless Interactive Networked Display (WIND) which is controlled by a central server (not in scope). It operates by creating a connection to the server and then handling requests to display content formatted as a PNG image.

Some content must be interactive which is enabled by the display being touch sensitive and the server sending an indexed coordinates map of touch-squares on the display. The device must be extremely low cost, so interaction is limited by the device mapping touch coordinates to the proper id and sending that id back to the server.

The device further has a speaker for supplying audio cues, which comprises a set of indexed predefined sounds. Each square on the touch-map can have an audio cue associated with it, specified by the index of the sound. The server can update the stored audio samples.

The device has four peripherals: the wireless transceiver, the audio peripheral, the touch display, and the proximity detector. All four are managed uniformly by the peripheral handler and report their readiness and failures on the Peripherals interface.

A proximity detector tells the device when a user is standing in front of it and when the user has left. The display is only made responsive when both a user is present and content is loaded. Updates are gated by the device: the server initiates each update with a request, and the device only accepts the request when no user is currently interacting with the displayed content. This guarantees that updates pushed by the server cannot land in the middle of an interaction. Presence transitions are also reported to the server (once content is loaded), so the server knows when it is worth retrying a previously rejected update request.

Server Connector

Context

Description

Controls and establishes the connection with the server. It waits for the device to become ready and then starts advertising its presence on the network to be discoverable by a server.

Behavior

The component is responsible for managing the lifecycle of the server connection over the wireless link.

Start-up

The component must not begin advertising until the device as a whole signals readiness. Activity received before that point is silently discarded.

Discovery

When ready, the component makes the device discoverable by broadcasting an advertisement packet every ADVERTISEMENT milliseconds. This continues until a server attempts to connect or the link is lost.

Connection request

A server may at any time during discovery request to connect, identifying itself with a serverId. The component must:

  • Stop advertising while the request is being evaluated.
  • If the server is not in the whitelist of permitted servers, reject the request via the radio so further requests from the same server are blocked, then resume discovery.
  • Otherwise, instruct the radio to establish the connection and wait at most CONNECTION_TIMEOUT milliseconds for confirmation.

If confirmation arrives in time, the device is considered connected to the identified server. If the timeout elapses first, the connection attempt is abandoned and discovery resumes.

Disconnection

If the radio reports that an established connection has been lost, the component must return to discovery so that a new server can find the device.

Error

If the device as a whole reports an unrecoverable error, the component must cancel any pending timers, disable the radio frontend, and return to its uninitialized condition. It must accept no further activity from the radio until the device signals readiness again.

Device Status Controller

Context

Description

Takes care of initializing the device by waiting for all the peripherals to be operational.

Behavior

The component is responsible for determining when the device as a whole is ready and for detecting peripheral failures.

Peripherals

The device has four peripherals: the wireless transceiver, the audio peripheral, the touch display, and the proximity detector. All four are brought up uniformly by PeripheralHandler:start() and signal readiness and failure on the Peripherals interface.

Start-up

On construction, the component starts the peripherals and begins counting how many of them have come up. The device must not be reported as ready until NUM_PERIPHERALS peripherals have signalled they are operational.

Each peripheral that becomes operational signals so; the component counts the signals and re-evaluates whether the full set has been received. As soon as the count reaches NUM_PERIPHERALS, the component announces device-wide readiness by emitting Device:ReadyInd. From that point on the device is considered operational.

Failure

If any peripheral reports failure, whether before or after the device has reached the operational condition, the component announces device-wide failure by emitting Device:ErrorInd and enters a terminal failed condition. While failed, the component is inert: it makes no further announcements, and recovery requires reconstruction of the component.

UI Controller

Context

Description

Displays content from the server, captures user interaction, and tracks user presence in front of the device.

Behavior

The component is responsible for displaying content received from the server, capturing the user's interaction with that content, reporting interaction back to the server, and reporting user presence to the server.

Start-up

The component must not present any content or accept any input until the device as a whole signals readiness. Once ready, the component initializes the audio peripheral, then the touch display. Each peripheral signals when it is itself ready; the audio peripheral additionally reports the number of stored sounds it can hold (numSounds), which the component records.

When the touch display is ready, it reports its pixel dimensions. The component forwards these dimensions to the server so that the server knows the canvas size for any content it pushes. The component is then ready to receive content.

Presence and responsiveness

The component tracks user presence in userPresent. The flag is updated unconditionally on every Proximity:UserDetectedInd and Proximity:UserLeftInd, regardless of what else the component is doing.

On top of this baseline, presence transitions are also reported to the server, but only once content has been loaded — the server does not care about presence before there is anything to interact with. Once content is loaded, every subsequent presence transition is reported, including transitions that happen while an update transaction is in progress.

The touch display is only made responsive when both a user is detected in front of the device and content is currently loaded. When the user leaves while interaction is enabled, the component makes the display unresponsive and notifies the server. The display is not cleared by the device — the server decides whether to push fresh content for the next user.

The proximity detector is contracted to emit UserDetectedInd and UserLeftInd strictly on transitions — never two UserDetectedInd in a row, nor two UserLeftInd in a row. The component therefore does not handle redundant presence events; receiving one would indicate a contract violation by the detector.

Update transaction

The server is in charge of sequencing content. To prevent the user from interacting with content that is in the process of being replaced, every update is initiated by the server as a request and must be explicitly accepted by the device before the server pushes any content.

Accepting or rejecting a request

On Content:StartUpdateReq, the component decides whether the update can be handled right now. The request is rejected with Content:StartUpdateRejected in any of the following circumstances:

  • A user is currently interacting with the content. The component remains available for interaction, and the server is expected to retry later — typically after it observes a sendUserLeft() notification.
  • An update transaction is already in progress. The server is contractually not expected to issue overlapping update requests, but the device rejects defensively.
  • The device is not yet ready to receive content, for example because the peripherals are still being initialized.

Otherwise, the component replies with Content:StartUpdateAccepted and disables the touch display so that no user touches are registered while content is being replaced.

Applying updates

After accepting, the component applies each Content:ContentInd received during the update by storing the new image, touch map, and sound map, and showing the new image on the display. It applies each Content:SoundInd by storing the indicated sound at the indicated index in the audio peripheral. Any number of either kind may be received in a single update.

Completion

On Content:UpdateDoneInd, content is now loaded. If a user is currently present, the component re-enables touch input and proceeds to interaction. Otherwise, it leaves touch input disabled and waits for the user to be detected.

It is the server's responsibility to follow this protocol — the device does not enforce a timeout on the update transaction. If the server fails to issue Content:UpdateDoneInd after the device has accepted, the component remains in the update transaction until a device-wide error or restart intervenes.

Interaction

While content is being displayed, the touch display is responsive, and the user may touch the screen. On a touch event identified by an (x, y) pixel coordinate, the component must:

  • Look up the touch area in the stored touch map and identify the corresponding sound from the stored sound map.
  • Play that sound through the audio peripheral.
  • Make the touch display unresponsive so that no further touches are registered before the server reacts.
  • Report the touched area to the server.

The component then awaits the server's next update request, which will typically push the next screen of content.

Error

If the device as a whole reports an unrecoverable error, the component must make the touch display unresponsive, show the default error screen, clear the local presence flag, and return to its uninitialized condition.

Handlers wrap external resources (timers, network endpoints, hardware peripherals) so the components can interact with them through interfaces and function calls.

Wireless Transceiver

Description

This handler takes care of receiving and transmitting data packets on the radio frontend by interfacing with the radio hardware peripheral. By looking at the message-type of incoming messages, indications are forwarded either to the Connection or Content interface.

The wireless transceiver is one of the device's peripherals: it is brought up by PeripheralHandler:start() together with the audio peripheral, the touch display, and the proximity detector, and it signals its readiness and any failure on the Peripherals interface so that DeviceStatusController can include it in its readiness count. Once readiness has been signalled, the radio is ready to advertise and accept connection requests.

Proximity Detector

Description

Interfaces with the proximity peripheral to detect when a user steps in front of the device and when they leave. Brought up by PeripheralHandler:start() along with the other peripherals; signals readiness via the Peripherals interface and emits presence events on the Proximity interface.

Contractually emits UserDetectedInd and UserLeftInd strictly on transitions: never two of the same kind in a row. The UiController relies on this contract and does not handle redundant events.
